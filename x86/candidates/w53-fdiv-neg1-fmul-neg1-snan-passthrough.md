# w53: `fdiv X, -1.0` lowers to bare `xorps` sign-bit toggle, preserving sNaN

**File:lines:**
- `llvm/lib/CodeGen/SelectionDAG/DAGCombiner.cpp:19609-19628` (visitFDIV: `(fdiv X, c2) -> (fmul X, 1/c2)`)
- `llvm/lib/CodeGen/SelectionDAG/DAGCombiner.cpp:19266-19272` (visitFMUL: `(fmul X, -1.0) -> (fsub -0.0, X)`)
- `llvm/lib/CodeGen/SelectionDAG/DAGCombiner.cpp:19052-19067` (visitFSUB: `(fsub -0.0, X) -> (fneg X)`)

## Reasoning

`visitFDIV` rewrites `fdiv X, -1.0` to `fmul X, 1/(-1.0)` = `fmul X, -1.0`
(reciprocal is exact). `visitFMUL` then rewrites `fmul X, -1.0` to
`fsub -0.0, X`. `visitFSUB` then matches the `fsub -0.0, X` pattern and
rewrites it to `fneg X`. None of the three folds requires `nnan`, so the chain
fires for plain IEEE-754 `fdiv`. On x86, FNEG lowers to `xorps`/`xorpd` of the
sign mask — pure bitwise sign flip.

The chain is observable as a different IR seed (`fdiv` vs `fmul`) than the
already-filed w12-fmul-neg1; both bottom out in the same FSUB → FNEG path that
the inline `// FIXME: This transform will change the sign of a NaN and the
behavior of a signaling NaN. It is only valid when a NoNaN flag is present.`
already documents at line 19057.

## Candidate IR

```ll
define double @fdiv_neg1(double %x) {
  %r = fdiv double %x, -1.0
  ret double %r
}
```

## x86 codegen

`llc -mtriple=x86_64-linux-gnu` (no fast-math flags on the IR):

```asm
fdiv_neg1:
  xorps .LCPI0_0(%rip), %xmm0     # mask = 0x8000000000000000 -- bare FNEG
  retq
```

No `divsd`, no rounding, no exception semantics. The codegen is identical to
a unary `fneg` despite the IR being `fdiv`.

## Confirmed wrong runtime behavior on x86_64

Driver:

```c
uint64_t snan_bits = 0x7FF4000000000000ULL;   // sNaN, mantissa MSB=0, lo set
double x; __builtin_memcpy(&x, &snan_bits, 8);
double r = fdiv_neg1(x);
uint64_t b; __builtin_memcpy(&b, &r, 8);
// b == 0xfff4000000000000   -- sign flipped, sNaN STILL signaling (bit 51 = 0)
```

Observed:
```
input       = 0x7ff4000000000000 (sNaN)
fdiv x,-1.0 = 0xfff4000000000000  (just a sign flip; bit 51 still 0 -> sNaN)
```

The sNaN bit pattern was passed through unchanged with only the sign bit
toggled. An actual `divsd` would have quieted the NaN (set bit 51 → result
`0xfffc000000000000`).

## Why this matters even though LangRef permits any NaN payload

LangRef does allow non-strict `fdiv` to return an unspecified NaN payload for
NaN operands. However:

1. **Strict variant doesn't fold.** The corresponding `llvm.experimental.constrained.fdiv` with `metadata !"fpexcept.strict"` correctly emits `divsd` and quiets sNaN. The strict ↔ non-strict divergence in observable bits is a behavior cliff that breaks programs migrating between the two.
2. **Same x86 instruction for `fneg` and `fdiv X,-1.0`.** The user wrote an arithmetic op; the compiler emitted a sign-bit toggle. Any consumer that inspects bit patterns (e.g. NaN-payload-carrying ABIs, JS NaN-boxing, GPU shader languages that round-trip through LLVM) will see the sNaN bit propagated.
3. **Same fold also affects `fdiv 1.0, ...` and `fdiv X, c` when 1/c is exactly representable** — the entire `(fdiv X, c2) -> (fmul X, 1/c2)` family pipes into the same `xorps` sink whenever `1/c2 == -1.0`.

## Coverage note relative to w12

w12-fmul-neg1-fsub-snan.md covers the FMUL seed.  This candidate is the FDIV
seed of the same FSUB→FNEG sink, so a code consumer importing IR generated by
a frontend that emits `fdiv X, -1.0` (rather than `fmul X, -1.0`) hits the same
bug from a different IR pattern. Both should be guarded by `Flags.hasNoNaNs()`
in `visitFSUB`'s `(fsub -0.0, N1) -> (fneg N1)` arm; fixing that one site cures
all upstream seeds.
