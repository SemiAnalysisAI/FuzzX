# m058: nibble-table byte-select high bit is miscompiled at `-O0`

Found while fuzzing upstream LLVM HEAD with llvm/llvm-project#198373,
llvm/llvm-project#196418, llvm/llvm-project#198412, and
llvm/llvm-project#198419 applied. The original oracle finding was:

```text
kind=oracle
index=7
input=0x1010101
o0=0x8000007F
o2=0x7F
expected=0x7F
```

Run the reproducer with:

```bash
known-miscompiles/run_ll_reproducer.sh \
  known-miscompiles/m058-nibble-bytesel-highbit/reduced.ll
```

Observed result on LLVM HEAD with the local PR patches:

```text
iteration=1
index=2
input=0x00000000
O0=0x8000007f
O2=0x0000007f
mismatch=true
```

## Reduction

The reduced kernel uses 256 zero inputs. The original full fuzzer input was
intermittent at lane 7, but the reduced LLVM HEAD testcase reproduces
deterministically at lane 2 in repeated standalone runs.

The surviving expression computes a one-iteration signed-overflow/masked-shift
loop value, feeds it through a generated nibble lookup table, then feeds the
packed nibble-table result through the byte-lane select idiom. The final store
is a funnel-shift-shaped expression:

```llvm
%fuzz.funnel.idiom.left = shl i32 %fuzz.bytesel.idiom.carry.xor, 31
%fuzz.funnel.idiom.right = lshr i32 255, 1
%fuzz.funnel.idiom.fshl.raw = or i32 %fuzz.funnel.idiom.left, %fuzz.funnel.idiom.right
```

In the original fuzzer finding, the LLVM interpreter oracle and `-O2` agree on
`0x0000007f`, while `-O0` stores `0x8000007f`. In the reduced standalone
testcase, the tested compilers all produce `-O0`/`-O2` disagreements on
zero-input lanes; the first mismatching lane can differ across toolchains.

## Root Cause Notes

The `-O0` lowering keeps the nibble-table and byte-select expression as a long
VALU sequence. Near the final store, it computes the byte-select carry with
several `v_bitop3_b32`, `v_cndmask_b32`, `v_lshl_or_b32`, and `v_add3_u32`
instructions, then shifts that low bit into bit 31:

```asm
v_bitop3_b32 v8, v8, s2, v10 bitop3:0x48
v_bitop3_b32 v2, v2, s2, v9 bitop3:0x48
v_lshl_add_u32 v2, v2, s4, v8
v_bitop3_b32 v4, v4, s2, v7 bitop3:0x48
v_bitop3_b32 v5, v5, s2, v6 bitop3:0x48
v_add3_u32 v2, v2, v4, v5
v_xor_b32_e64 v2, v2, v3
v_lshl_or_b32 v2, v2, s0, v3
```

On LLVM HEAD, the `-O2` lowering computes the same source expression with a
shorter SDWA byte-select sequence and stores `0x0000007f` for the first
mismatching reduced lane. This points at a lowering bug in the nibble-table /
byte-select high-bit path rather than an IR semantics issue.

## Fuzzer Follow-Up

The fuzzer now rejects generated byte-lane select carry values derived from the
generated nibble-table pack by default. Set
`FUZZX_ALLOW_M058_NIBBLE_BYTESEL_HIGHBIT=1` to re-enable this bug class.
